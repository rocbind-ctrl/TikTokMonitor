# 优化方案

## 结论

当前程序已经可以运行和打包，但适合作为“恢复版基线”。如果要给团队长期使用，建议按下面顺序优化。

## P0：稳定性和安全

- 把服务端固定为唯一写数据库的进程，客户端只通过 API 操作。
- 增加数据库定时备份和恢复说明。
- 为同步任务增加队列状态、失败重试和人工重跑入口。
- 将默认密码和 API Key 强制改为部署时配置。
- 给远程访问增加 HTTPS/VPN 部署说明。

## P1：API 和客户端体验

- 统一 JSON 返回格式，减少客户端兼容成本。
- 给账号、视频、告警、同步日志增加分页和搜索参数。
- 桌面端补齐账号详情、视频详情、批量导入、设置页。
- 同步按钮增加排队中、同步中、完成、失败状态。
- 告警中心增加筛选、批量已读和跳转到账号/视频。

## P2：性能

- 对账号列表和视频列表增加分页，避免数据量大时一次性加载。
- 为常用过滤字段增加数据库索引。
- 将大批量同步拆成后台任务，并限制并发数。
- 将耗时 provider 调用和 Web 请求线程隔离。
- 数据量上来后迁移到 PostgreSQL。

## P3：工程化

- 增加 API 单元测试和端到端 smoke test。
- 用 GitHub Actions 生成三平台桌面安装包。
- 增加版本号、变更日志和 release 流程。
- 后续再考虑代码签名、自动更新和角色权限。

## 建议执行顺序

1. 先完成本次 monorepo、API、Tauri、CI 基础迁移。
2. 再完善分页、详情页、批量导入和设置页。
3. 最后做 PostgreSQL、权限、自动更新和代码签名。
